# Manifesto

Software é construído por pessoas.

Pessoas tomam decisões.

Decisões possuem contexto.

Contexto gera conhecimento.

Conhecimento não deve depender da memória.

Por esse motivo existe a Project Knowledge Base.

Ela preserva a história do Melody Sync para que nenhuma decisão importante seja perdida ao longo da evolução do projeto.

---

# Objetivo

A Project Knowledge Base é a memória oficial do Melody Sync.

Seu objetivo é registrar todas as ideias, pesquisas, descobertas, decisões, problemas, aprendizados e planejamentos realizados durante o desenvolvimento do projeto.

Ela não substitui a documentação oficial.

Seu papel é preservar o contexto das decisões para evitar perda de conhecimento, retrabalho e repetição de discussões.

---

# Filosofia

O Melody Sync é desenvolvido utilizando a metodologia **Knowledge Driven Development (KDD)**.

Isso significa que toda implementação deve nascer de conhecimento previamente registrado.

A Knowledge Base representa esse conhecimento.

Enquanto a documentação oficial descreve **o que foi decidido**, a Knowledge Base registra **como chegamos até essa decisão**.

---

# Objetivos da Knowledge Base

- Preservar o histórico do projeto.
- Centralizar pesquisas.
- Registrar decisões importantes.
- Organizar ideias.
- Evitar repetição de discussões.
- Facilitar futuras implementações.
- Servir como referência para novos colaboradores.
- Alimentar a documentação oficial.

# Relação com a documentação oficial

A Knowledge Base é utilizada como fonte para a criação dos documentos oficiais.

Exemplo:

Research
→ Project Charter

Research
→ Product DNA

Problems
→ Requirements

Decisions
→ ADR

Ideas
→ Architecture

Architecture
→ Código

Dessa forma, nenhum documento oficial nasce sem contexto.

---

# Convenções

Toda informação registrada deve:

- possuir contexto;
- possuir data quando relevante;
- registrar o motivo da decisão;
- ser escrita de forma objetiva;
- evitar duplicação de conteúdo;
- referenciar documentos relacionados sempre que possível.

---

# O que deve ser registrado

Exemplos de informações que pertencem à Knowledge Base:

- pesquisas de mercado;
- análises de concorrentes;
- problemas encontrados em ferramentas existentes;
- descobertas durante o desenvolvimento;
- ideias para funcionalidades futuras;
- decisões técnicas;
- dúvidas ainda não respondidas;
- hipóteses;
- aprendizados obtidos durante o projeto.

---

# O que NÃO deve ser registrado

Não fazem parte da Knowledge Base:

- documentação de API;
- documentação de código;
- tutoriais de utilização;
- documentação de instalação;
- documentação de arquitetura final;
- especificações já consolidadas.

Esses conteúdos pertencem à documentação oficial do projeto.
